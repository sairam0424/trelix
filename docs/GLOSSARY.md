# trelix Glossary

Alphabetical reference for terms used throughout trelix's source code, APIs, and documentation. Version: v2.7.0.

---

**BM25**
The Okapi BM25 probabilistic ranking function (Robertson & Zaragoza, 2009) used by trelix for keyword-based symbol search. BM25 scores a document by term frequency (TF) saturated by parameter k1 (penalises repetition), document length normalised by b, and inverse document frequency (IDF) weighting rarer terms more heavily. trelix drives BM25 through SQLite's built-in FTS5 virtual table (`symbols_fts`) which is created with `tokenize='porter ascii'` and updated by INSERT/UPDATE/DELETE triggers that keep it in sync with the `symbols` table. FTS5's native `bm25()` function returns a negative score (closer to zero means better match); trelix converts this to a positive `score = 1.0 / (1.0 + abs(bm25_rank))` before feeding it into RRF fusion. The BM25 retrieval leg is enabled by default (`top_k_bm25=20`) and is the `"bm25"` source tag visible in `SearchResult.source`.

**Blast Radius**
The set of all symbols, files, and callers that would be affected if a given symbol were changed or removed. In trelix, blast radius analysis is a first-class retrieval intent (`IntentType.BLAST_RADIUS`) handled by the standard pipeline with `import_direction="reverse"` — the import graph expander traverses *incoming* edges (files that import the target) rather than outgoing ones. The retrieval planner also seeds import-path patterns (e.g., `@aliases`) via `seed_from_import_paths()` to surface indirect dependents that reference the symbol through barrel exports or path aliases. The MCP tool `trelix-blast-radius` and the CLI intent `blast_radius` expose this analysis as a structured query. Call-graph callers are included via `expand_with_call_graph`, which follows caller edges (not just callees) for the blast radius intent.

**Call Graph**
A directed graph of caller-to-callee relationships extracted from source files by tree-sitter during indexing. Each call edge is stored as a `CallEdge` row in the `calls` table (`caller_id`, `callee_name`, `callee_id`, `line`, `callee_type_hint`). During index finalization, trelix performs a second-pass resolution that populates `callee_id` by matching `callee_name` against known `symbols.qualified_name` entries — preferring type-hint-scoped matches (e.g., `callee_type_hint="UserService"` narrows candidates to `UserService.*` symbols) to reduce false-positive edges. The `Retriever` public API exposes `get_callers(symbol_name)` (1-hop incoming edges) and `get_callees(symbol_name)` (1-hop resolved outgoing edges, internal only); both return `list[SearchResult]` sorted by file path and line number for determinism. Unresolved calls to external libraries or stdlib remain in the table with `callee_id=None` and are excluded from graph traversal.

**Chunk**
The unit of text that gets embedded into the vector store and indexed for BM25 search. Each `Chunk` dataclass (fields: `symbol_id`, `chunk_text`, `token_count`, `embedding`, `id`) corresponds to one `Symbol` — trelix uses whole-symbol chunking by default, meaning every function, class, or method becomes exactly one chunk. The `chunk_text` is not raw source: the `Chunker` prepends a structured context header (`# File: ... | Language: ...`, `# Imports: ...`, `# Class: ...`) so the embedding model understands the symbol's position in the codebase without seeing the whole file. Token count is computed via `tiktoken` (`cl100k_base` encoding) and stored at index time; chunks exceeding `max_tokens_per_chunk` (default 512) are truncated to preserve the header and signature while cutting the body from the bottom. Sub-symbol chunks at block or statement granularity are available as an optional 6th retrieval leg (see: MultiGranularity Sub-chunk Indexing).

**Context Budget**
The maximum number of tokens (`context_token_budget`, default 12,000) that the `ContextAssembler` is allowed to fill when building the `RetrievedContext.context_text` block passed to the LLM. After RRF fusion, graph expansion, and optional reranking, the assembler greedily packs `SearchResult` chunks in score order, stopping when adding the next chunk would exceed the budget. Each chunk's `token_count` (pre-computed at index time via tiktoken) is used for fast budget accounting without re-tokenising at query time. The `RetrievedContext.total_tokens` and the `budget_pct` field in the debug trace show how much of the budget was consumed. The budget is configurable via `TRELIX_RETRIEVAL_CONTEXT_TOKEN_BUDGET` and is distinct from `synthesis_max_tokens`, which caps the LLM's reply.

**Contextual Chunking**
An optional indexing mode (`ChunkerConfig.contextual=True`) where an LLM generates a 2–3 sentence natural-language summary of each symbol and prepends it to the `chunk_text` before embedding and BM25 indexing. This technique, based on Anthropic's contextual retrieval research, enriches each chunk with high-level intent that the raw code may not express — reducing retrieval failure by approximately 67% on ambiguous natural-language queries. The `ContextualChunker` class extends `Chunker` and calls the configured LLM with a fixed prompt (`"In 2-3 sentences, describe what this code does..."`) for each symbol's body; failures degrade silently to the base chunker. The generated summary is persisted to `symbols.context_summary` in SQLite and is included in the FTS5 index so it improves both vector and BM25 recall. Enable with `TRELIX_CHUNKER_CONTEXTUAL=true` (requires LLM API access).

**DimensionGuard**
A startup safety check that detects embedding dimension mismatches before they corrupt query results. When a user switches embedding providers (e.g., from local 384-dim to Azure 3072-dim), the existing HNSW index contains vectors of the wrong dimension; without detection, sqlite-vec returns wrong results or raises cryptic errors. `DimensionGuard.check(db, current_dimension, provider)` reads the stored dimension from `index_metadata` and raises `DimensionMismatchError` if there is a mismatch; it is called during both `Retriever` and `Indexer` initialization. `DimensionMismatchError` carries the stored and current dimension values and prints a clear migration hint. The fix is `trelix migrate-vectors --reset ./your-repo` (which calls `DimensionGuard.reset()` to clear the stored dimension) followed by a fresh `trelix index`; `--reset` deletes all stored embeddings and must be followed by re-indexing.

**DiffHunk**
A single changed block in a unified diff, produced by `DiffParser.parse()`. The `DiffHunk` dataclass carries: `file_path` (relative path from the `+++ b/` header), `old_start` and `new_start` (1-indexed line numbers from the `@@ -a +b @@` header), `old_lines` and `new_lines` (hunk size), `added` (lines starting with `+`), `removed` (lines starting with `-`), and `context` (unchanged surrounding lines). `DiffHunk.to_search_query()` extracts identifiers from the added and removed lines to build a retrieval query suitable for passing to the `Retriever`. `DiffParser.from_git(repo_path, base, head)` runs `git diff --unified=3` and parses its output into a list of `DiffHunk` objects.

**DiffReviewer**
A retrieval-augmented code reviewer for git diffs. `DiffReviewer.review(hunks)` iterates over `DiffHunk` objects and for each hunk: (1) builds a retrieval query via `DiffHunk.to_search_query()`, (2) retrieves relevant codebase context via the `Retriever`, (3) calls the LLM with the hunk diff text plus context capped at 3,000 tokens, and (4) parses the JSON response into `ReviewComment` objects (fields: `file_path`, `line_start`, `line_end`, `severity`, `comment`). The LLM is instructed to return a JSON array of comments with `INFO | WARN | ERROR` severity; empty array means no issues. Any hunk-level failure is caught and logged as a non-fatal warning so the reviewer continues to the next hunk. The MCP server exposes this functionality via the `review_diff` tool.

**ExpandResult**
The return type of `MultiQueryExpander.expand()` — a frozen dataclass introduced in v2.4.0 as the single observability surface for multi-query expansion. Fields: `queries` (list of query strings: original + up to N LLM-generated variants), `llm_used` (bool — `False` when the LLM was unavailable or expansion failed), and `elapsed_ms` (wall-clock time of the LLM call, 0.0 on fallback). The `Retriever` stores the `ExpandResult` in thread-local storage (`_trace_local.expand_result`) so it can be forwarded to `TelemetryWriter.record()` without threading it through all call frames. When `llm_used=False`, `queries` always contains exactly the original query, making the pipeline's behavior identical to running without expansion enabled.

**Federated Search**
Cross-repository search that fans out a single query to multiple independently-indexed trelix repositories in parallel and merges results via RRF. `FederatedRetriever` reads the `RepoRegistry`, spawns one `Retriever` per repo in a `ThreadPoolExecutor` (bounded by `max_workers`, default 4), collects up to `k` results per repo, applies RRF fusion across all per-repo lists, and deduplicates by `(file_path, symbol_id)`. The `trelix search-all` CLI command and `TRELIX_FEDERATION_ENABLED=true` environment variable activate federation at retrieval time. Per-repo failures are caught and logged as warnings so a single unavailable repo does not block the entire query. Each repo in the registry can carry a `weight` multiplier (default 1.0) that scales its results' contribution to the merged RRF score.

**FederatedRetriever Cache**
A thread-safe in-memory TTL cache on `FederatedRetriever` that avoids re-querying all repos for identical queries. The cache key is a SHA-256 hash of the concatenation of `query`, sorted registered repo paths, and `k` — meaning any change to the query text, the repo set, or the requested result count produces a cache miss. Entries expire after `cache_ttl` seconds (default 120.0); `cache_ttl=0` disables caching entirely. `cache_stats()` returns `{"hits": int, "misses": int, "size": int}` for observability. `clear_cache()` evicts all entries immediately, for example after a repo is re-indexed. The cache is per-`FederatedRetriever` instance, not global, so parallel sessions each maintain their own cache.

**FLARE**
A confidence-gated re-retrieval loop inspired by Forward-Looking Active REtrieval Augmented Generation (Jiang et al., EMNLP 2023, arXiv:2305.06983). When `flare_enabled=True`, `FLARELoop.run(query)` performs an initial retrieve-then-synthesize pass; if the generated answer contains uncertainty phrases ("I don't know", "cannot find", "insufficient context", etc.), it enriches the query with `"(focus on implementation details and concrete code)"` and re-retrieves up to `flare_max_retries` times (range 1–3, default 1; the old env var `TRELIX_RETRIEVAL_FLARE_MAX_ITER` was renamed to `TRELIX_RETRIEVAL_FLARE_MAX_RETRIES` in v2.4.0 and the new field enforces `le=3`). Each retry runs the full pipeline from scratch (new retrieval, new synthesis). The heuristic is simpler than the original paper's token log-probability monitoring but achieves similar intent: detect low-confidence answers and attempt to ground them with additional retrieved context. When `flare_enabled=False` (the default), `FLARELoop` is a transparent pass-through.

**HyDE**
Hypothetical Document Embeddings (Gao et al. 2022, arXiv:2212.10496) — a zero-shot technique that improves vector retrieval by embedding a synthetic code snippet rather than the raw natural-language query. The `HyDEExpander` calls the LLM with a fixed system prompt ("write a SHORT hypothetical code snippet (3-8 lines) that would directly answer [query]") and uses the returned snippet as the text to embed. The intuition is that a hypothetical code snippet occupies the same dense embedding neighbourhood as real code that answers the query, whereas a natural-language question occupies a different neighbourhood. The QueryPlanner generates HyDE snippets for each sub-query and stores them in `SubQuery.hyde_snippet`; if empty, the `Retriever` uses `semantic_query` as the fallback embed text. The optional `TRELIX_RETRIEVAL_HYDE_FALLBACK=true` flag enables HyDE on the retrieval path even when the planner did not generate a snippet.

**Import Graph**
A directed graph of file-to-file import relationships extracted from `import` / `require` / `use` statements during indexing and stored in the `imports` table (`file_id`, `imported_from`, `imported_names`). After all files are indexed, the `Indexer` runs a second-pass `resolve_import_file_ids()` step that matches `imported_from` module paths against indexed `files.rel_path` entries (using language-specific heuristics) to populate `imports.imported_file_id` with concrete database IDs. The retrieval pipeline uses the resolved import graph for two purposes: forward expansion (surface files that this file depends on, for `feature_flow` queries) and reverse expansion (surface files that depend on this file, for `blast_radius` queries). Wildcard imports (`import *`) are stored with `imported_names=["*"]`. Cross-language imports (e.g., a TypeScript file importing a Go module) are stored but `imported_file_id` remains unresolved.

**Index**
The primary artifact of a `trelix index` run: a SQLite database at `.trelix/index.db` relative to the repo root. The schema (see `db.py` DDL) contains eight main tables: `files` (one row per indexed file, with SHA-256 content hash for incremental updates), `symbols` (one row per extracted symbol), `calls` (directed call graph edges), `type_edges` (inheritance and interface implementation edges), `imports` (file-level import edges), `chunks` (one row per embeddable text chunk, 1:1 with symbols by default), `sub_chunks` (optional block/statement granularity chunks), and `file_summaries` (optional RAPTOR-style per-file summaries). The FTS5 virtual table `symbols_fts` is automatically kept in sync with `symbols` via triggers and provides BM25 keyword search over `name`, `qualified_name`, `docstring`, `body`, and `context_summary`. The `chunk_embeddings` are stored in a separate sqlite-vec extension table (or in an external Qdrant/LanceDB store). The `graph_metadata` table stores per-symbol PageRank centrality and community assignment computed by the CodeGraph builder.

**LanceDB**
An optional alternative vector store backend for large-scale deployments, activated by `TRELIX_STORE_BACKEND=lance`. `LanceVectorStore` wraps the `lancedb` Python library and stores all chunk embeddings in a LanceDB table (`chunks` by default) at a configurable URI (`LANCE_URI`, default `.trelix/lance`). LanceDB 0.6+ provides ARM-native HNSW with zero SQLite dependency and is validated at 3–5x faster insert throughput vs. sqlite-vec HNSW at >100k vectors (vecdb-bench measurements). It is the recommended backend for repos with >500k chunks (approximately >2M lines of code) or for Apple Silicon and ARM server deployments. Install with `pip install trelix[lance]`; the backend selection is the only config change required — all other pipeline behavior is identical.

**MCP**
Model Context Protocol — a standardised JSON-RPC interface that allows AI coding assistants (Claude, Cursor, Copilot, etc.) to call trelix tools directly from their context window. The `trelix-mcp` package exposes a `FastMCP` server with six tools: `search_code` (hybrid search with cursor-based pagination), `index_codebase` (initial or incremental index), `get_symbol` (fetch full symbol source by qualified name), `get_callers` (1-hop incoming call graph), `get_callees` (1-hop outgoing call graph), and `review_diff` (RAG-based diff review). Two MCP Resources (`trelix://index/stats`, `trelix://repo/{repo_path}/manifest`) and two Prompts (`trelix-search`, `trelix-explain`, `trelix-blast-radius`) complete the server's surface area. The server is launched with `uvx trelix-mcp` or via the `trelix mcp` CLI command and communicates over stdio (default) or HTTP.

**MultiQueryExpander**
An LLM-based query diversification component that generates N rephrased variants of the primary query to increase retrieval recall on varied vocabulary. When `multi_query_enabled=True` (env: `TRELIX_RETRIEVAL_MULTI_QUERY=true`), the `Retriever` instantiates a `MultiQueryExpander` with `n=multi_query_count` (default 2) and calls `expander.expand(primary_query)` after the primary sub-query legs run. Each non-duplicate variant is wrapped into a minimal `SubQuery` object (semantic_query only, no HyDE or grep hints) and run through all retrieval legs in a parallel `ThreadPoolExecutor`. The resulting per-variant leg buckets are merged into the main `leg_results_list` before RRF fusion, effectively adding retrieval legs 7+ to the standard 6-leg pipeline. Returns an `ExpandResult`; when the LLM is unavailable, `queries=[original]` and `llm_used=False` so the pipeline continues without expansion.

**MultiRepoWatcher**
A v2.4.0 addition that watches all repos registered in `RepoRegistry` for filesystem changes simultaneously, triggering incremental re-indexing on modification. Unlike `FileWatcher` (which uses `watchdog` and monitors a single repo), `MultiRepoWatcher` uses `watchfiles.awatch()` with all registered repo paths in a single async call; the Rust-layer debounce (default 1,600ms) collapses rapid saves into single re-index events. A hash guard (`MD5` of file bytes, cached in `_file_hashes`) prevents cascade loops when the indexer writes `.trelix/` files to the source tree. Deleted files are removed from the SQLite index and vector store via `db.delete_file_by_path()`. Activated with `trelix watch-all`; requires `pip install trelix[watch]` (the `watchfiles` package).

**PageRank Boost**
An optional post-rerank score multiplier that elevates symbols with high import-graph centrality in the final result ranking. When `pagerank_boost_enabled=True` (env: `TRELIX_RETRIEVAL_PAGERANK_BOOST=true`), the `Retriever` calls `get_top_central_symbols(db, top_n=200)` to retrieve the 200 symbols with the highest PageRank centrality scores (computed by `CodeGraph` during indexing and persisted to `graph_metadata.centrality`), then multiplies their RRF scores by `pagerank_boost_factor` (default 1.3, configurable 1.0–3.0 via `TRELIX_RETRIEVAL_PAGERANK_BOOST_FACTOR`). This surfaces widely-imported hub symbols (e.g., a `models.py` module imported by 50 other files) higher than their raw retrieval rank would place them. The boost is applied after reranking and before context assembly to avoid disrupting the reranker's semantic relevance signal.

**Provider**
The embedding backend used to convert symbol text into dense float vectors. Configured via `EmbedderConfig.provider` (env: `TRELIX_EMBEDDER_PROVIDER`). trelix ships nine providers: `local` (sentence-transformers `all-MiniLM-L6-v2`, 384-dim, no API key, default), `openai` (`text-embedding-3-large`, 3072-dim, requires `OPENAI_API_KEY`), `azure` (Azure OpenAI `text-embedding-3-large`, 3072-dim, requires `AZURE_API_KEY` + `AZURE_ENDPOINT`), `voyage` (`voyage-code-3`, 1024-dim, supports Matryoshka output dimensions, requires `VOYAGE_API_KEY`), `local-code` (Salesforce `SFR-Embedding-Code-2B_R`, 4096-dim, no API key), `bge-code` (BAAI `bge-code-v1`, 768-dim, CoIR SOTA 2025, requires `pip install trelix[bge-code]`), `nomic-code` (nomic-ai `CodeRankEmbed`, 768-dim, requires `pip install trelix[local]`), `bedrock-titan` (Amazon Titan v2, 1024-dim), and `bedrock-cohere` (Cohere English v3, 1024-dim). Switching providers after indexing requires running `trelix migrate-vectors --reset` to clear dimension-mismatched embeddings (see: DimensionGuard).

**Qdrant**
An optional cloud-ready vector store backend for distributed and large-scale deployments (>500k chunks), activated by `TRELIX_STORE_BACKEND=qdrant`. `QdrantVectorStore` connects to a Qdrant service at `QDRANT_URL` (default `http://localhost:6333`) using `QDRANT_API_KEY` for authentication and stores embeddings in collection `QDRANT_COLLECTION` (default `trelix`). The collection is created automatically on first use with HNSW parameters `m=16` and `ef_construct=200` for high-recall approximate nearest neighbour search at high cardinality. Install with `pip install trelix[qdrant]`. Unlike the default sqlite-vec backend, Qdrant persists vectors independently of the SQLite metadata store, enabling shared vector indexes across multiple trelix processes or hosts.

**QueryTelemetry**
An optional per-query observability record written to the `query_telemetry` table in the SQLite index. When `telemetry_enabled=True` (off by default), `TelemetryWriter.record(context, elapsed_ms, expansion_result)` appends one row per `retrieve()` call containing: `query` text, `intent` (from the QueryPlanner), `elapsed_ms` (wall-clock time from query start to context assembly completion), `result_count`, and (v2.4.0 addition) three expansion columns: `expansion_used` (bool), `expansion_variants` (int — number of queries in `ExpandResult.queries`), and `expansion_elapsed_ms` (LLM latency for the expansion call). The telemetry table drives `trelix eval` which computes nDCG@10 against a golden query set and is the primary tool for measuring retrieval improvement over time. `TelemetryWriter` never raises — all failures are logged at DEBUG level and swallowed.

**RRF (Reciprocal Rank Fusion)**
The score fusion algorithm used throughout trelix to combine ranked result lists from heterogeneous retrieval legs into a single merged ranking. The formula is: `score(doc) = Σ 1 / (k + rank_i)` summed over all lists where the document appears, with `k=60` (Cormack et al. 2009 standard constant). RRF only uses rank position (not raw scores), making it robust to non-comparable score scales across BM25 (range 0.05–0.5), cosine distance (range 0.7–0.95), and grep hits (fixed score). In trelix the up-to-six standard legs (vector, BM25, grep, file summary, sub-chunk, sparse) plus any multi-query variant legs are all fused in `reciprocal_rank_fusion(ranked_lists, k=60, weights=file_type_weights)`. The optional `weights` dict applies a per-language multiplier to RRF scores after accumulation (e.g., `markdown=0.3` downranks documentation relative to source code). `FederatedRetriever` uses the same function to merge results across repos.

**RepoRegistry**
The persistence layer for federated search configuration, stored as JSON at `~/.config/trelix/repos.json` (overridable by `.trelix/federation.json` in any repo). Each entry is a `RepoEntry` dataclass with `alias` (human-readable name), `path` (absolute path to the repo root), and `weight` (RRF score multiplier, default 1.0). The `RepoRegistry` class provides `add(alias, path, weight)`, `remove(alias)`, `list()`, and `save()` methods; `add()` raises `ValueError` on duplicate alias. Managed via `trelix repo add/remove/list` CLI commands. The registry is loaded at `FederatedRetriever` construction time and at `MultiRepoWatcher` startup; an empty registry is valid and produces an empty result set without error.

**Retriever**
The primary query-time orchestrator that converts a natural-language or keyword query into a `RetrievedContext` object ready for LLM injection. The pipeline: (1) `QueryPlanner.plan(query)` — LLM intent classification and sub-query decomposition (falls back to `default_plan()` when no API key is configured); (2) intent routing — `file_overview`, `project_overview`, `config_lookup` take direct DB paths skipping retrieval legs; (3) standard path — three to six retrieval legs (vector, BM25, grep, optional file summary, sub-chunk, sparse) run per sub-query, optionally in parallel for multi-sub-query plans; (4) optional multi-query expansion; (5) `reciprocal_rank_fusion` across all leg buckets; (6) graph expansion (call graph + import graph + type edges); (7) optional cross-encoder reranking; (8) optional PageRank boost; (9) context assembly within `context_token_budget`. Every query writes a structured debug trace JSON to `.trelix/debug/`. Instantiated via `Retriever(config)` where `config` is an `IndexConfig`.

**SearchResult**
The fundamental result object that flows through the entire retrieval pipeline, from individual leg outputs through RRF fusion, graph expansion, reranking, and final context assembly. The `SearchResult` dataclass carries: `chunk` (the `Chunk` object with `chunk_text` and pre-computed `token_count`), `symbol` (the `Symbol` with `qualified_name`, `kind`, `line_start`, `line_end`, `signature`, `body`, `docstring`), `file` (the `IndexedFile` with `rel_path` and `language`), `score` (float, interpreted as RRF score after fusion or raw cosine/BM25 score before), `rank` (1-indexed position in the current list), and `source` (string tag: `"vector"`, `"bm25"`, `"grep"`, `"graph_expansion"`, `"file_summary"`, `"sub_chunk"`, `"sparse"`, `"file_direct"`, `"graph_callers"`, `"graph_callees"`, `"graph_importers"`). Immutability is enforced by creating new `SearchResult` instances (spread pattern) when modifying `score` or `rank` in the pipeline.

**Symbol**
A named code construct extracted from a source file by tree-sitter during indexing. The `Symbol` dataclass fields are: `file_id`, `name` (bare identifier), `qualified_name` (dot-separated path, e.g., `LoginView.authenticate_user`), `kind` (one of `SymbolKind`: `function`, `method`, `class`, `interface`, `struct`, `enum`, `constant`, `variable`, `module`, `section`), `line_start` / `line_end` (1-indexed, inclusive), `signature` (e.g., `def authenticate_user(self, username: str) -> User`), `body` (full verbatim source text of the symbol), `docstring`, `decorators`, `is_public`, `parent_id` (links methods back to their enclosing class), and `context_summary` (LLM-generated summary when contextual chunking is enabled). The Indexer produces one `Symbol` per language-specific extractor match; every `Symbol` gets exactly one `Chunk`.

**Tree-sitter**
The incremental AST parser library that drives all symbol extraction in trelix. Tree-sitter parses source files into concrete syntax trees using per-language grammars (WASM bindings via `tree-sitter` Python package) and language-specific node-visitor classes called extractors. trelix ships 20+ language extractors covering Python, JavaScript, TypeScript, TSX, Go, Rust, Java, Kotlin, Ruby, C, C++, C#, Razor, CSHTML, CSPROJ, Markdown, JSON, YAML, TOML, HTML, and CSS. Each extractor walks the tree-sitter AST and emits `Symbol`, `CallEdge`, `TypeEdge`, and `ImportEdge` objects without regex or line-based heuristics. Tree-sitter's incremental re-parsing capability is not currently used by trelix's watcher (which re-parses the entire changed file), but the symbol-level granularity it provides (exact line ranges and signatures) is fundamental to the context-header quality in `Chunk` objects.

**Vector Store**
The component that stores and retrieves dense float embeddings for approximate nearest neighbour (ANN) search. The default backend is `SQLiteVectorStore` (sqlite-vec extension), which creates a `vec0` virtual table with optional HNSW index (`TRELIX_STORE_HNSW=true`, default). HNSW parameters are configurable (`hnsw_m=16`, `hnsw_ef_construction=200`, `hnsw_ef_search=50`); sqlite-vec >= 0.1.6 is required for HNSW support, with fallback to flat scan on older versions. Alternative backends are `QdrantVectorStore` (cloud-ready, `TRELIX_STORE_BACKEND=qdrant`) and `LanceVectorStore` (ARM-native, `TRELIX_STORE_BACKEND=lance`); all three implement the `BaseVectorStore` ABC with the same interface (`upsert_batch`, `search`, `delete_batch`, `count`, and specialised methods for file-summary and sub-chunk embeddings). The active backend is determined at `Retriever` / `Indexer` construction by `make_vector_store(config, dimension)`.

**Watch / Watch-All**
Two complementary filesystem monitoring modes for incremental index maintenance. `trelix watch` uses `FileWatcher` (watchdog-based, synchronous) to monitor a single repository; it debounces rapid saves to the same file (default 500ms window) and triggers `Indexer.index_file()` on modifications or `db.delete_file_by_path()` on deletions. `trelix watch-all` uses `MultiRepoWatcher` (watchfiles-based, async) to monitor all repos registered in `RepoRegistry` simultaneously with a single `watchfiles.awatch()` call over all paths; debounce is handled by watchfiles' Rust layer (default 1,600ms) and a hash guard prevents cascade re-indexing when the indexer writes to the `.trelix/` directory inside the watched tree. Both modes require the optional `watch` extra (`pip install trelix[watch]`); `watch` depends on `watchdog>=4.0.0` while `watch-all` depends on `watchfiles>=0.21`. Neither mode re-indexes files whose SHA-256 (watch) or MD5 (watch-all) content hash has not changed.
