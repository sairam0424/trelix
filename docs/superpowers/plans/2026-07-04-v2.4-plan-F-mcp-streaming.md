# v2.4 Plan F — MCP Progress Notifications + Pagination (Streaming-Adjacent)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add two streaming-adjacent UX improvements to `trelix-mcp`: (1) progress notifications during `index_codebase` (long-running) via MCP's `ctx.report_progress()`, and (2) cursor-based pagination in `search_code` for large result sets — since the MCP spec does NOT support streaming tool responses (tools always return in a single message).

**Architecture:** FastMCP `Context` is injected into tool functions via `ctx: Context` parameter. `index_codebase` calls `ctx.report_progress(current, total)` during indexing stages. `search_code` gains `cursor: int = 0` (offset) and returns a `next_cursor` field so callers can page through results. This is the MCP-spec-approved pattern for large payloads.

**Research-grounded facts:**
- The MCP protocol spec (2025) does NOT support streaming tool responses — tools return in a single `tools/call` response.
- FastMCP 3.x does not expose `resources/subscribe` at the developer level — bypassing it requires low-level mcp SDK, which is fragile and unsupported.
- `ctx.report_progress(current, total)` is the correct FastMCP API for progress (sends `notifications/progress` to the client).
- Cursor-based pagination via a `next_cursor` response field is the MCP-spec-approved pattern for large resource lists.

**Tech Stack:** Python 3.11+, `fastmcp>=2.0.0`, `mcp>=1.0.0`.

@mcp.tool()
def search_code(
    query: str,
    repo_path: str,
    k: int = 10,
    cursor: int = 0,
) -> dict:
    """
    Search the indexed codebase using natural language queries.

    ⚠️ IMPORTANT:
    - repo_path must be an ABSOLUTE path to an already-indexed repository.
    - Run index_codebase first if you receive an error about a missing index.

    🎯 When to Use:
    - Find specific functions, classes, or implementations
    - Understand architecture before making changes
    - Locate all callers of a function before refactoring
    - Find similar patterns to follow when adding code

    📄 Pagination:
    - Use cursor=0 for first page (default).
    - If next_cursor is not null, pass it as cursor for the next page.
    - k controls page size.

    Returns:
        {"results": [...], "next_cursor": int|null, "total_available": int}
    """
    _log.info("search_code query=%r repo=%s k=%d cursor=%d", query, repo_path, k, cursor)
    config = IndexConfig(repo_path=repo_path)
    # Fetch k * 5 to enable pagination without re-querying
    fetch_k = max(k * 5, cursor + k * 2)
    ctx = Retriever(config).retrieve(query)
    all_results = ctx.results

    page = all_results[cursor : cursor + k]
    next_cursor = cursor + k if cursor + k < len(all_results) else None

    return {
        "results": [
            {
                "file": r.file.rel_path,
                "symbol": r.symbol.qualified_name,
                "kind": r.symbol.kind.value,
                "lines": f"{r.symbol.line_start}-{r.symbol.line_end}",
                "score": round(r.score, 4),
                "source": r.source,
                "body": r.symbol.body[:800],
                "language": r.file.language.value,
            }
            for r in page
        ],
        "next_cursor": next_cursor,
        "total_available": len(all_results),
    }
```

@mcp.tool()
def index_codebase(
    repo_path: str,
    provider: str = "local",
    ctx: Context | None = None,
) -> dict:
    """
    Index a repository for code search. Run once before calling search_code.

    ⚠️ IMPORTANT:
    - Stores the index in <repo_path>/.trelix/index.db (zero external infra).
    - Re-run to refresh after large code changes; incremental update is fast.

    ✨ Providers:
    - local   — no API key, CPU-only, fast for small repos
    - openai  — requires OPENAI_API_KEY, best quality
    - azure   — requires AZURE_API_KEY + AZURE_ENDPOINT
    - voyage  — requires VOYAGE_API_KEY, best code-specific quality

    Progress notifications are sent if the MCP client supports them.
    """
    _log.info("index_codebase repo=%s provider=%s", repo_path, provider)

    from typing import Literal, cast
    config = IndexConfig(
        repo_path=repo_path,
        embedder=EmbedderConfig(
            provider=cast(
                Literal["openai", "azure", "local", "voyage", "local-code"],
                provider,
            )
        ),
    )

    # Send progress notifications if client supports it
    if ctx is not None:
        try:
            import asyncio
            asyncio.get_event_loop().run_until_complete(
                ctx.report_progress(0, 3)  # stage 0/3: starting
            )
        except Exception:
            pass  # progress is best-effort; never block indexing

    indexer = Indexer(config)
    stats = indexer.index()

    if ctx is not None:
        try:
            import asyncio
            asyncio.get_event_loop().run_until_complete(
                ctx.report_progress(3, 3)  # stage 3/3: done
            )
        except Exception:
            pass

    return stats
```

**Note:** FastMCP handles the async/sync bridge for `ctx.report_progress()`. If `run_until_complete` fails (e.g., event loop is already running in async context), the progress notification is silently skipped — indexing always succeeds.
