# trelix v2.4.0 — World Release Readiness Report

**Date:** 2026-07-05  **Version:** 2.4.0  **Verdict:** CONDITIONAL
**Overall Score:** 81/100  **Confidence:** MEDIUM

## Executive Summary
Trelix v2.4.0 ships a technically strong core: 1,508 tests at 100% pass rate, strict mypy clean, ruff-clean, all 6 feature planes fully exercised, MCP toolset intact at 6 tools, and an excellent first-run user journey. The core `trelix` package and `trelix-mcp` are correctly versioned, built, and functional.

However, the release is structurally incomplete in three ways that prevent a clean GO. First, two of three integration packages — `trelix-langchain` and `trelix-llama-index` — remain at version 2.0.0 in both their `pyproject.toml` files and on PyPI. The release workflow's `skip-existing: true` silently no-oped the publish step because the 2.0.0 artifacts already existed; no 2.4.0 build was uploaded. Any user attempting `pip install trelix-langchain==2.4.0` receives HTTP 404. This is a broken release surface for a meaningful segment of the user base.

Second, an undocumented breaking startup constraint is present: the renamed `flare_max_retries` field carries a new `le=3` ceiling that the old `flare_max_iterations` did not enforce. Users who previously set `TRELIX_RETRIEVAL_FLARE_MAX_ITER=5` or higher will hit a pydantic `ValidationError` on process startup immediately after upgrade, with no migration path documented in the CHANGELOG. The CHANGELOG mentions the rename but omits the range constraint entirely — a silent production cliff for a subset of existing deployments.

Third, the CHANGELOG contains a data integrity defect: the entire v2.4.0 breaking-change block (pagination envelope + migration code) was copy-pasted verbatim into the v2.3.0 section header at lines 65–122. Version 2.3.0 had zero breaking changes (all features were opt-in flags). This corrupts the historical record and will mislead integrators who rely on CHANGELOG for upgrade planning.

A minor pipeline gap — no Linux binary in the release matrix — affects standalone binary users on the dominant server platform but is lower urgency since `pip install trelix` remains functional for Linux.

These four items are all pre-release fixable with under a day of engineering work. No security vulnerabilities were found, no data-loss paths exist, and the adversarial score of 62 reflects packaging and documentation completeness failures rather than runtime correctness bugs. The release should not proceed until the integration packages and the undocumented constraint are resolved; the CHANGELOG and Linux binary are strongly recommended before tagging public availability.

## Verdict: CONDITIONAL
Hold the public announcement and PyPI promotion of v2.4.0 as a 'full release' until the three REQUIRED pre-release actions are completed. The engineering work is estimated at 2–4 hours: bump two pyproject.toml files, re-trigger the release workflow, add two sentences to the CHANGELOG, and delete the erroneous CHANGELOG section. Once those are done and `pip install trelix-langchain==2.4.0` resolves successfully, this release is GO. The core package is genuinely release-quality; the blockers are packaging hygiene and documentation completeness, not runtime correctness. Do not issue the public launch post or send release notification emails until PyPI shows all three packages at 2.4.0 and the CHANGELOG correction is on the default branch.

## Pre-Release Actions
- REQUIRED — Bump trelix-langchain/pyproject.toml and trelix-llama-index/pyproject.toml from version 2.0.0 to 2.4.0, update any __version__ attributes in their source packages, re-run the release workflow so both packages are built and published to PyPI under 2.4.0. Verify with `pip index versions trelix-langchain` and `pip index versions trelix-llama-index` that 2.4.0 appears before tagging the release complete.
- REQUIRED — Add an explicit migration note to the CHANGELOG v2.4.0 `flare_max_retries` section documenting the new hard upper bound: 'Note: the field now enforces ge=1, le=3. If you previously set TRELIX_RETRIEVAL_FLARE_MAX_ITER to a value greater than 3, you must lower it to 3 or lower before upgrading, or the process will raise a pydantic ValidationError at startup.'
- REQUIRED — Fix the CHANGELOG v2.3.0 section: delete the erroneous copy-pasted v2.4.0 breaking-change block that appears at lines 65–122 (the second [2.3.0] header block). Version 2.3.0 was a zero-breaking-change release; the section should begin directly with the Embedding Dimension Guard and Multi-Query Retrieval Wiring content that currently appears at line 123.
- RECOMMENDED — Add `ubuntu-latest` to the binary build matrix in the release workflow so a Linux x64 standalone binary is built and attached to the GitHub Release alongside the existing macOS ARM64 and Windows x64 artifacts. Linux is the dominant server/CI platform and the only one currently without a binary distribution path.

## Post-Release Monitoring
- Monitor PyPI download counts for trelix-langchain 2.4.0 and trelix-llama-index 2.4.0 in the first 48 hours to confirm the packages are being discovered and installed — anomalously low numbers relative to trelix-mcp would suggest discovery or dependency pinning issues.
- Track GitHub Issues and Discussions for reports of 'ValidationError' or startup crashes in the first 72 hours post-release; any spike signals that users with FLARE_MAX_ITER > 3 are hitting the undocumented constraint ceiling.
- Watch for MCP client compatibility reports around the search_code pagination envelope breaking change — this is the one documented breaking change in v2.4.0 and the most likely source of integration failures from early adopters pinning to the old list[dict] return type.
- Monitor the GitHub Release asset download counts to confirm macOS ARM64 and Windows x64 binaries are being fetched; a disproportionate absence of Linux binary downloads should prompt expedited Linux binary release.
- Check PyPI for any user-submitted bug reports or questions about trelix-langchain/trelix-llama-index dependency resolution against the trelix>=0.4.0 pin — these packages should have their dependency floor updated to trelix>=2.4.0 to prevent accidental installation against incompatible older core versions.

## Important Findings
- [package_metadata] packages/trelix-langchain/pyproject.toml declares version = "2.0.0" but the world release is v2.4.0
- [package_metadata] packages/trelix-llama-index/pyproject.toml declares version = "2.0.0" but the world release is v2.4.0
- [release_pipeline] trelix-langchain and trelix-llama-index pyproject.toml versions are still 2.0.0 — never bumped to 2.4.0. The release workflow ran `python -m build` on stale 2.0.0 source, then `skip-existing: true` silently no-oped the PyPI publish step. Both packages remain at 2.0.0 on PyPI; 2.4.0 returns HTTP 404.
- [release_pipeline] The binary build matrix only covers macos-latest (arm64) and windows-latest (x64). There is no ubuntu-latest entry, so no Linux binary is built or attached to the GitHub Release. Linux users have no standalone binary for v2.4.0.
- [adversarial] trelix-langchain and trelix-llama-index package versions are frozen at 2.0.0 — both pyproject.toml files and __version__ attributes were not bumped to 2.4.0. They will be published to PyPI under v2.0.0 by the release workflow, not v2.4.0, creating a version mismatch visible on PyPI.
- [adversarial] Users who previously set TRELIX_RETRIEVAL_FLARE_MAX_ITER to any value greater than 3 (e.g. 5, 10) will get an unhandled pydantic ValidationError on startup after upgrading to v2.4.0. The CHANGELOG migration guidance only mentions the rename, not the new constraint (Le(le=3)). This is an undocumented silent startup cliff.
- [adversarial] The CHANGELOG v2.3.0 section (line 123) is a verbatim copy-paste of the v2.4.0 breaking-change block, including the line 'After (v2.4.0): search_code returns a pagination envelope'. This is factually wrong — v2.3.0 search_code returned list[dict] (confirmed via git show v2.3.0). The pagination was introduced in v2.4.0 commit 596519b.
